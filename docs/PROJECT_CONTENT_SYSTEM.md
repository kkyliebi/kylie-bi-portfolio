# Project Content System

**Status:** 🌳 Core structure / 🌿 content

This document defines how portfolio projects move from scattered evidence into accurate, publishable case studies.

## Purpose

The project system must support two goals at once:

1. publish a credible portfolio quickly enough to begin job applications;
2. preserve a structure that can grow into a deeper long-term archive.

The first release does not require every project to be equally complete. It requires a coherent archive, three strong complete cases, and honest handling of incomplete evidence.

## Source-of-truth hierarchy

When project information conflicts, use this order:

1. verified project facts approved by Kylie;
2. original project files, presentations, images and written records;
3. structured content in `src/content`;
4. supporting documentation in `docs`;
5. generated placeholder narrative;
6. chat recollection or inference.

Generated prose must never be treated as historical evidence.

## Canonical project set

The current archive contains five project routes:

1. TYRANNO
2. The Infinite Discussion
3. Audi Integrated Brand Communication
4. Deear Magazine
5. Madam, in Eden, I'm Adam

This order is the current editorial sequence. Changing the set or order is a publication decision and requires explicit discussion.

## Shared case-study anatomy

Every project uses the same six chapters:

1. **Context** — what situation, tension or opportunity initiated the work;
2. **Research** — what evidence, references or observations shaped the direction;
3. **Concept Development** — how the central idea and communication logic emerged;
4. **Process** — how the idea was tested, produced, revised and coordinated;
5. **Final Outcome** — what was made and how the parts operate together;
6. **Reflection** — what the project reveals about Kylie's method, judgement and future practice.

The chapter structure is stable. The amount of content may differ by project.

## Required project record

A project may be marked `verified` only when the following factual fields are supported:

- title;
- year or date range;
- project context;
- Kylie's role;
- collaborators or team context, where relevant;
- intended audience;
- actual deliverables;
- final outcome;
- publication or confidentiality status.

A project may remain public as `draft` only when the language is clearly non-deceptive and does not invent facts.

## Editorial fields

Each project should contain:

- **Title** — public project name;
- **Subtitle** — optional factual descriptor;
- **Category** — professional framing, not a list of software;
- **Role** — Kylie's contribution in industry-readable language;
- **Lead question** — the unresolved question that invites entry;
- **Summary** — a compact explanation of the project and its relevance;
- **Six chapters** — evidence-led case-study narrative;
- **Media** — hero, research, process, archive and gallery material;
- **Next project** — editorial transition within the archive.

## Evidence rules

### Acceptable evidence

- original presentation decks;
- exported project PDFs;
- photographs and final artwork;
- process files and sketches;
- written briefs;
- production documents;
- credits and collaborator records;
- Kylie's factual confirmation.

### Evidence is not established by

- a filename alone;
- generic placeholder copy;
- a category label;
- visual inference without supporting context;
- model-generated reconstruction.

### Writing rule

Every specific factual claim should be traceable to evidence. Where evidence is incomplete, use narrower language rather than filling the gap.

## Media responsibilities

- **Hero** establishes the project's visual identity immediately.
- **Research** shows what informed the work rather than decorating the page.
- **Process** demonstrates judgement, iteration and production responsibility.
- **Archive** preserves source material, fragments and contextual evidence.
- **Gallery** presents resolved outputs and relationships between deliverables.

Media selection should prove the narrative. It should not merely fill layout slots.

## Completion levels

### Level 0 — Configured

The route, title, category, lead question and chapter slots exist.

### Level 1 — Reconstructed draft

A coherent draft exists, but factual gaps remain and production assets may be missing.

### Level 2 — Fact verified

Core metadata and narrative claims have been checked against evidence or confirmed by Kylie.

### Level 3 — Production ready

Verified copy, selected media, captions, alt text, credits and responsive asset variants are complete.

### Level 4 — Published and reviewed

The case is live, checked in context and revised using real reader or application feedback.

## Minimum viable release

The job-application release should contain:

- a complete Landing, About, Archive and Contact journey;
- all five projects visible in the archive;
- three projects at Level 3;
- the remaining projects at no less than an honest Level 1;
- no invented role, client, impact, metric or outcome;
- clear contact and CV access.

The three production-priority projects should collectively prove:

1. concept and visual communication thinking;
2. creative production and system-building ability;
3. relevance to at least one target employment direction.

## Division of responsibility

### Repository maintainer

- maintain structure, naming and cross-links;
- extract reusable facts from available evidence;
- draft and edit project narratives;
- identify contradictions and missing proof;
- prepare grouped decision requests;
- keep `src/content` and documentation aligned.

### Kylie

Kylie's input is required only for:

- factual correction;
- authorship and role boundaries;
- confidentiality;
- collaborator credits;
- final project selection and order;
- meaningful creative-direction choices;
- publication approval.

Routine repository maintenance should not be delegated to Kylie.

## Change protocol

When changing a verified project record:

1. state what changed;
2. state why it changed;
3. identify the new evidence or decision;
4. update all affected content and documentation;
5. avoid creating a parallel version unless historical preservation is useful.
